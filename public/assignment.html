Semantic versioning is a method of keeping records of software releases. It is the set of rules that dictate how the version number are assign and incremented. The semantic version consists of three numbers separated by decimals MAJOR_VERSION.MINOR_VERSION.PATCH_VERSION. We will see it in the real world like this 1.8.3. It indicates the following library is in its major version 1, minor version 8, and patch version 3.

  • Patch version

    According to the documentation, It should only update while backward compatibility bugs introduced. Version incremented while encountering a bug that needs to fix quickly.

  • Minor Version

    The minor version incremented while adding a new feature in the library. For example, the slider library's version is 1.8.3 library added a new animation feature in it and the version will be 1.9.3. Likewise, another reason to increment the minor version is while depreciating or removing the old feature.

  • Major Version

    The major version increases when breaking changes introduced in public API, for example re-writing the entire library. After releasing the major all the minor and patch versions are initialized. If the library is upgraded from 1.9.3 the new version will be 2.0.0.

Simplistically it's like maintaining accounts of all the changes you've done while developing software. This version may seem like an arbitrary number it provides in-depth details about the library. Nowadays without it, handling dependencies is impossible. 

public class Stack {
	private Object[] elements;
	private int size = 0;
	private static final int DEFAULT_INITIAL_CAPACITY = 16;
	public Stack() {
		elements = new Object[DEFAULT_INITIAL_CAPACITY];
	}
	public void push(Object e) {
		ensureCapacity();
		elements[size++] = e;
	}
	public Object pop() {
		if (size == 0)
			throw new EmptyStackException();
		return elements[--size];
	}
	private void ensureCapacity() {
		if (elements.length == size)
			elements = Arrays.copyOf(elements, 2 * size + 1);
	}
}

The above code has an issue of memory leak. In the pop method, it's not clearing the reference object only reducing the size of an array. If we push 100 objects in a stack it won't be any issue related to a memory leak. However, if we pop 50 objects and push 10 objects there will be stale references, it will prevent garbage collection and in the end, there will be a memory leak of 40 objects. In order to prevent memory leaks in the stack we have to nullify or eliminate the obsolescent reference.

Solution
public Object pop() {
	if (size == 0) throw new EmptyStackException();
	elements[size] = null;
	return elements[--size];
}

In terms of the relational database model, a NULL indicates an unknown value. But this unknown value does not equivalent to zero value or a field that contains spaces. Due to the structure of the NULL values, it's not possible to use the operators like (=, <, > and <>) in the queries.The SQL standards using the WHERE clause as above will return the empty result sets. Using IS NULL and IS NOT NULL conditions will return match records.

Include the NULL field
SELECT * FROM table
WHERE field IS NULL;
Exclude the NULL field
SELECT * FROM table
WHERE field IS NOT NULL;

Race conditions are the type of programming error while multiple processes or threads trying to access or modify the shared data objects, shared resources, and shared memory. For example, a thread is trying to rename a file and another thread is trying to delete the same file. In this kind of condition, the second thread cannot delete the file because the file's name has already been renamed. On the other hand, if the file is deleted by the second thread, the first cannot rename cause it cannot find the file in the filesystem.

Semaphore and Mutual exclusion(mutex) are two approaches instituted to prevent race conditions. Due to some drawbacks of semaphore like priority inversion, non-practical method for largescale application cause leads to loss of modularity. Mutex seems to be a perfect solution for the critical section.

Because of the single-thread nature of nodejs, it may seem race conditions cannot occur. However, to prevent blocking event loop, it processes intensive operation in a thread pool where the multiple logical are scheduled concurrently.

Encryption is a practice of scrambling information in a way that only someone with a corresponding key unscrambles and reads it. Encryption is a two-factor function. It needs to be decrypted after encryption. For example, while logging we enter the password and the server decrypts the password saved in a database and compares it with the user's password.

On the other hand, hashing is the practice of using an algorithm to map data of any size to a fixed length. For example, the server saves generated JWT(JSON Web Token) in a database and compares it with the browser's token. The key difference between encryption and hashing is that hashing is not a two-factor function.

const crypto = require('crypto');

const generateSalt = (rounds = 12) => {
	if (rounds >= 15) throw new Error(`Rounds must be less than 15 `);
	if (typeof rounds !== 'number') throw new Error('Rounds must be a number');
	return crypto
	.randomBytes(Math.ceil(rounds / 2))
	.toString('hex')
	.slice(0, rounds);
}

const hasher = (password, salt) => {
	const hash = crypto
		.createHmac('sha512', salt)
		.update(password)
		.digest('hex');

	return {
		hash,
		salt,
		password
	}
}

const result = hasher('test1234', generateSalt());

function perimeter(length, breadth) {
if (!length || !breadth) {
	throw Exception(‘Undefined’);
}
if (length is not Number){
	throw Exception(‘Not a number’);
}
if (breadth is not Number){
	throw Exception(‘Not a number’);
}
if (length < 0 OR breadth < 0){
	throw Exception(‘dimension cannot be negative’);
}
return 2 * (length + breadth);
}
